# Dogfood Report — بوابة الطلبات (multi-page) · v2.3.0

> الاختبار الذي يحسم المؤجّلات: **هل sidebar/app-shell + data-app header حاجبان؟** بُني تطبيق متعدّد الصفحات حقيقي (تفاعلي) للإجابة بالأدلة، لا بالرأي. المعاينة: artifact `193004a4` · المصدر [`playground/request-portal/`](playground/request-portal/).

## 1. السؤال والجواب الحاسم
**هل sidebar/app-shell + app-header حاجبان لتطبيق متعدّد الصفحات؟ → نعم، قطعًا.**
- بمجرد وجود **أكثر من صفحة** (لوحة · طلبات · تفاصيل · طلب جديد) لزم **تنقّل دائم** (sidebar) و**سياق علوي** (app-header: عنوان الصفحة + بحث + حساب). لا يمكن بناء التطبيق بدونهما — `layout.md` كان يصف **النمط** فقط بلا spec مكوّن.
- لذلك رُقِّيا من BACKLOG وصِيغا قبل البناء (document-before-use)، ثم بُني الـshell عليهما.
- **القرار:** sidebar + app-header **مكوّنان أساسيان** لأي تطبيق متعدّد الصفحات (لا «nice to have»). بقية المؤجّلات (textarea/radio/switch/breadcrumb/dark-mode) **لم** تُثبت حاجبة هنا — تبقى مؤجّلة.

## 2. ما بُني
- **2 spec جديد:** [`sidebar.md`](design-system/components/sidebar.md) · [`app-header.md`](design-system/components/app-header.md) (المكوّنات 29→31).
- **shell تفاعلي + 4 صفحات** بتنقّل JS (vanilla، بلا أداة بناء): لوحة (KPI cards) · الطلبات (جدول+فلترة+بحث) · تفاصيل الطلب (detail-view + موافقة/رفض) · طلب جديد (نموذج).

## 3. ماذا نجح
- **التنقّل متعدّد الصفحات** يعمل: sidebar يحدّث النشط (`aria-current`) + عنوان الهيدر؛ بحث الهيدر العام يقفز للطلبات.
- **إعادة استخدام المكتبة بلا إعادة بناء:** table · filter-bar · status-badge · avatar · card(KPI) · detail-view(dl) · form · modal · empty-state · toast — كلها ركّبت الصفحات الأربع من النظام مباشرة. **دليل أن الأساس الإجباري قابل للتوسّع لتطبيق كامل.**
- **Anti-Generic عمليًا:** كل صفحة بتركيب مختلف حسب محتواها (لوحة KPI ≠ جدول ≠ تفصيل dl ≠ نموذج) — لا «قالب واحد لكل الصفحات».
- **المعالجة الحيّة:** موافقة/رفض عبر modal تُحدّث الحالة + الشارة + toast، وتُعطّل الأزرار بعد المعالجة.
- **البوّابات الست محفوظة:** صفر arbitrary · **صفر inline style خام** (حُوِّلت كلها utilities) · RTL منطقي · native-control reset · select بسهم مخصّص · ghost icon buttons في الهيدر · sidebar نشط بخلفية+لون لا حدّ · modal a11y §6.5/§6.6 · focus-visible فقط.

## 4. ما ظهر ضعيفًا / ملاحظات
- **الموبايل (درج التنقّل):** بُني بسيطًا (transform + scrim). يعمل، لكن لم أضف focus-trap كاملًا داخل درج الموبايل (الـsidebar كـnav لا dialog) — مقبول للعرض، يوثَّق كتحسين.
- **breadcrumb:** صفحة التفاصيل تستخدم زرّ «عودة» بدل breadcrump — كافٍ هنا (مستويان فقط)؛ لو تعمّقت المسارات لزم [breadcrumb](design-system/components/BACKLOG.md) (يبقى مؤجّلًا، لم يُثبت حاجبًا بعد).
- **textarea:** استُخدم في «طلب جديد» بقواعد input (كما يقول input.md) — يعمل، لكن لا spec مستقل (auto-resize/counter) → **High في BACKLOG، لم يُثبت حاجبًا** (نموذج بسيط).
- **dashboard:** KPI داخل card كافٍ؛ لا حاجة لمكوّن KPI مستقل الآن.
- loading/error للصفحة: لا backend لمحاكاتهما (موثّق).

## 5. هل احتجنا spec مؤجّلًا؟
- **نعم (حاجب):** sidebar · app-header → صِيغا واعتُمدا.
- **لا (غير حاجب هنا):** textarea (عمل بقواعد input) · breadcrumb (زرّ عودة كفى) · radio/switch/dark-mode (لم تُستدعَ).

## 6. الحكم
**v2.3.0 يثبت أن design-os يتوسّع من صفحة واحدة إلى تطبيق متعدّد الصفحات كامل** — بـshell من مكوّنَين جديدين + إعادة استخدام المكتبة بالكامل، مع حفظ البوّابات الست. **السؤال المفتوح الأكبر حُسِم:** sidebar + app-header أساسيان (مُنجزان الآن)؛ والباقي يبقى مؤجّلًا بأمان حتى يُثبت اختبارٌ لاحق حاجته.

## 7. التالي المقترح
- تصنيف BACKLOG بعد هذا الاختبار: **textarea** يرتقي لـ«High التالي» (يُستخدم فعلًا في النماذج)؛ **breadcrumb** «متوسط» (يُستدعى عند مسارات أعمق)؛ radio/switch/dark-mode/data-viz يبقون «حسب الحاجة».
- إن أردت: dogfood على **نموذج طويل** أو **لوحة بيانات بеё رسوم** لاختبار textarea/data-viz، أو الانتقال لاستخدام البوابة على بيانات حقيقية.
